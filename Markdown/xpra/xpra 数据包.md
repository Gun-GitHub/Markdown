# xpra 数据包

```
 xpra服务器和客户端之间交换的每个数据包都必须遵循完全相同的格式：一个8字节的头，后面跟着使用rencodeplus编码的数据包数据，可以选择使用lz4压缩。

只有当连接使用无法识别的协议时，xpra才能以不带任何数据包标头的纯文本错误消息进行响应。
```

- ## 框架
  
  当通过WebSocket、SSL、SSH或QUIC进行连接时，传输层显然会添加自己的帧。
  
  关于这些网络协议中的每一个所使用的分组成帧的信息，请参考这些协议。
  ```
   这里只记录了xpra自己的应用层协议。
  ```
- ## 数据包标头
  
  数据包标头由8个字节组成：
  |索引|类型|型长度（以字节为单位）|内容|
  |--|--|--|--|
  |0|Byte|1|P|
  |1|Byte|1|协议标志|
  |2|Byte|1|压缩级别|
  |3|Byte|1|块索引|
  |4|Long|4|有效载荷大小|
  |**协议标志**||||
  |协议标志是一个8位的位掩码值。||||
  
  它必须包含值16，用于rencodeplus数据包。其他值不再支持。
  
  这个值可以与以下值进行或运算：
  - 8，设置flush标志，通知数据包层后面没有其他数据包
  - 2，设置cipher标志，用于==AES加密的数据包==
    ```
    如果您能够安全地将AES密钥分发给每个客户端，请使用此选项。这在某种程度上类似于使用自签名证书的SSL模式。
    
    Xpra的AES加密层使用python cryptography库以AES-256 - 高级加密标准在CBC - 密码块链接模式、GCM - Galois/Counter模式、CTR - 计数器模式或CFB - 密码反馈模式下加密网络数据包。
    
    加密密钥可以存储在密钥文件中，或者使用keydata套接字选项指定。如果两者都不存在，并且使用了认证模块，则密码将被用作密钥数据。 密钥数据使用PBKDF2（基于密码的密钥派生函数2）进行扩展。 使用Python的os.urandom()生成的盐值适合用于加密使用。
    ```
    
    注意事项：
    - 也可以运行在AES-128或AES-192模式，但不推荐
    - HTML5客户端目前不支持GCM模式：Xpra-org/xpra-html5#94
    - 旧的服务器和客户端只支持CBC模式
    - 设置AES的逐步说明，请展开：
      
      AES使用示例 生成一个密钥：
      ```sh
      uuidgen > ./key.txt
      ```
      
      启动服务器：
      ```sh
      xpra start --start=xterm \
                 --bind-tcp=0.0.0.0:10000,encryption=AES,keyfile=key.txt
      ```
      
      客户端：
      ```sh
      xpra attach "tcp://localhost:10000/?encryption=AES&keyfile=./key.txt"
      ```
      
      **模式 **
    从版本4.3开始，客户端可以指定要使用的确切AES加密模式：
      ```sh
      encryption=AES-GCM。
      ```
      
      **旧语法 **
      ```
      在版本4.1之前，加密是全局配置的，适用于所有TCP套接字，使用以下语法：
      ```
      ```纯文本
      xpra start --start=xterm \
                  --bind-tcp=0.0.0.0:10000 \
                   --tcp-encryption=AES --tcp-encryption-keyfile=key.txt
      xpra attach tcp://$HOST:10000 --tcp-encryption=AES --tcp-encryption-keyfile=./key.txt` 
      ```
    - 指定密钥数据:
      
      keydata
      ```
      在较新的版本中，除了使用keyfile选项外，还可以在bind和attach字符串中内联keydata值：
      ```
      - keydata=0x... 用于十六进制编码的密钥
      - keydata=base64:... 用于base64编码的密钥
      - keydata=... 用于明文密钥
    
    一个主要的缺点是密钥数据可能会在进程列表中泄露。 然而，在某些情况下，生成不需要额外文件即可运行的命令可能更容易。
    - 调试
      
      要验证您的客户端连接是否使用了AES，请寻找cipher=AES：
      ```sh
      xpra info | grep cipher=
      ```
      
      要启用调试，请使用 ==-d crypto== 调试日志选项。
      
      **压缩等级**
      ```
      如果压缩等级为零，则负载未被压缩。或者至少，在网络层未被压缩：像素和音频数据可能仍然被压缩，但这是由它们各自的子系统处理，而不是传输层。
      
      这个位掩码值用于指示数据包负载是如何被压缩的。
      ```
  
  低4位表示压缩等级。高4位表示使用了哪种压缩器：
  - 16代表lz4
  - 64代表brotli
    
    **块索引**

块索引用于发送大型负载，并绕过数据包编码器。

  数据包块通常不使用数据包编码器或压缩器。

  接收方必须替换主数据包中的块索引处找到的项。

```
      发送一个假设的数据包（"example-large-packet", "foo", 20MB数据）的例子，发送2个块以获得更好的性能：
```

- 使用块索引2发送20MB数据（基于零）
- 发送（"example-large-packet", "foo", ""）与块索引0（0是主数据包）
接收方必须从这两个块重新组装原始数据包。
- ## 负载
  
  主负载的块索引为零。
  
  一旦根据需要按照压缩等级标志解压缩，就必须根据协议标志使用rencodeplus进行解码。
  
  它由一个项目列表组成。该列表中的第一个项目是数据包类型字符串。
  
  数据包类型可以跟随可变数量的可选参数
  
  **数据包类型**
  
  数据包类型是一个用于将数据包分派给正确处理程序的字符串。
  - 每个子系统应该为其所有的数据包类型使用相同的前缀。
    
    \*\*子系统 \*\*
    
    此文档涉及到单个协议功能，它链接到每个子系统的实现和技术文档。
    
    大多数模块是可选的，请参阅安全考虑因素。
    
    **概念**
    - 客户端模块：由客户端加载的功能实现，它与服务器端的相应“客户端连接模块”接口
    - 客户端连接模块：对于每个到客户端的连接，服务器将实例化一个处理程序
    - 服务器模块：由服务器实现的功能，它可能与多个“客户端连接模块”交互
      |子系统|客户端模块|服务器模块|客户端连接模块|用户文档|
      |--|--|--|--|--|
      |音频|audio|audio|audio|音频特性|
      |剪贴板|clipboard|clipboard|clipboard|剪贴板特性|
      |MMAP|mmap|mmap|mmap|自动启用|
      |日志|remote-logging|logging|none|日志使用|
      |通知|notifications|logging|notification|通知特性|
      
        每个子系统都设计为增强整体应用程序的功能，提供特定领域的服务和特性。
    
    		例如，音频子系统负责处理音频数据的传输，剪贴板子系统允许在客户端和服务器之间共享剪贴板内容。MMAP子系统提高了数据传输的效率，而日志和通知子系统则为用户提供了重要的反馈信息和系统状态更新。

			  一旦对等方处理了hello数据包，如果hello数据包包含一个别名能力，其中包含从数据包类型到整数的映射，那么数据包类型也可以作为一个整数发送。

##   **参数**

  应该使用的唯一数据类型是：

- 整数
- 布尔值
- 字典
- 可迭代对象：列表和元组
- 字节数组
- 字符串

可以编码浮点数和None值，但应避免使用。

## Hello 数据包(握手数据包)

Hello 数据包是作为握手的初始数据包。直到双方都接收到hello数据包，连接才算完全建立。

Hello数据包包含一个参数，这个参数是一个字典，包含对等方宣传的所有能力。

例如，预期的hello数据包应该包含一个版本能力，其中包含版本字符串。 它们也可能包括一个用户名值，可用于认证。 每个子系统还将添加其自己的属性。
